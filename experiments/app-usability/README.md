# การทดลองที่ 2: ทดสอบทั้งแอปกับผู้ใช้จริง (Controlled Usability Experiment)

การทดลองนี้วัดว่า **แอป MoneyMind ช่วยให้คนบันทึกเงินได้ดีกว่าวิธีเดิมจริงไหม** โดยเทียบกับการจดในสเปรดชีต/สมุด — เป็นการทดลองแบบมีกลุ่มควบคุม วัดผลเป็นตัวเลข ทำสถิติได้ เอาเข้ารูปเล่มโครงงานได้ทันที

> การทดลองที่ 1 (โฟลเดอร์ `experiments/`) วัด **ความแม่นของ AI**
> การทดลองที่ 2 (โฟลเดอร์นี้) วัด **ประสิทธิภาพของแอปเมื่อคนใช้จริง**
> สองอันนี้เสริมกัน: อันแรกพิสูจน์ "สมองแม่น" อันที่สองพิสูจน์ "ใช้แล้วดีขึ้นจริง"

---

## 1. กรอบวิธีการทางวิทยาศาสตร์

| หัวข้อ | เนื้อหา |
|---|---|
| **ปัญหา** | วัยรุ่น/คนทั่วไปไม่ค่อยจดบันทึกรายรับ-รายจ่าย เพราะวิธีเดิม (สมุด/สเปรดชีต) ช้าและยุ่งยาก แอปที่ออกแบบมาช่วยได้จริงหรือไม่ วัดอย่างไร |
| **สมมติฐาน** | ผู้ใช้บันทึกรายการด้วยแอป MoneyMind **เร็วกว่า ผิดพลาดน้อยกว่า และพึงพอใจมากกว่า** การจดด้วยสเปรดชีต/สมุด |
| **ตัวแปรต้น** | วิธีบันทึกข้อมูล (แอป MoneyMind ↔ สเปรดชีต/สมุด) |
| **ตัวแปรตาม** | (1) เวลาที่ใช้บันทึกครบชุด (วินาที) · (2) จำนวนข้อผิดพลาด (รายการที่บันทึกผิด/ตกหล่น) · (3) คะแนนความพึงพอใจ SUS (0–100) |
| **ตัวแปรควบคุม** | ชุดรายการทดสอบชุดเดียวกัน 10 รายการ · ผู้เข้าร่วมคนเดียวกันทำทั้งสองวิธี (within-subject) · อุปกรณ์/สถานที่/แสง/เวลาใกล้เคียงกัน · ผู้จับเวลาคนเดิม |
| **กลุ่มตัวอย่าง** | เพื่อน/คนรู้จัก แนะนำ **≥ 15 คน** (ขั้นต่ำ 10 เพื่อให้สถิติมีน้ำหนัก) |

**จุดที่ทำให้ "เป็นวิทยาศาสตร์" จริง ๆ (เขียนเน้นในเล่ม):**
- ใช้แบบ **within-subject** (คนเดียวทำทั้ง 2 วิธี) → ตัดความต่างระหว่างบุคคลออก ทำให้เทียบกันแฟร์
- **สลับลำดับ (counterbalance)**: คนเลขคี่เริ่มด้วยแอปก่อน คนเลขคู่เริ่มด้วยสเปรดชีตก่อน → กัน "learning effect" (พอทำรอบสองจะเร็วขึ้นเองไม่ว่าวิธีไหน)
- ใช้ **SUS (System Usability Scale)** ซึ่งเป็นแบบวัดความใช้ง่ายมาตรฐานสากล 10 ข้อ → อ้างอิงได้ ไม่ใช่ถามลอย ๆ

## 3. ขั้นตอนการทดลอง (ต่อผู้เข้าร่วม 1 คน)

1. บันทึกหมายเลขผู้เข้าร่วม (participant id) และลำดับ (คี่ = แอปก่อน / คู่ = สเปรดชีตก่อน)
2. อธิบายวิธีใช้เครื่องมือแรกสั้น ๆ 30 วินาที (เท่ากันทั้งสองเครื่องมือ)
3. **เริ่มจับเวลา** → ให้บันทึกครบ 10 รายการด้วยเครื่องมือแรก → **หยุดเวลา** จดวินาทีลง CSV
4. ตรวจผลที่บันทึก เทียบกับเฉลย → นับจำนวนรายการที่ผิด/ตกหล่น = error
5. พัก 1–2 นาที
6. ทำซ้ำข้อ 3–4 ด้วยเครื่องมือที่สอง
7. ถ้าเครื่องมือใดคือแอป → ให้ตอบแบบสอบถาม **SUS 10 ข้อ** (`sus-th.md`) กรอกคะแนน 1–5 ลง CSV ช่อง `sus_1`–`sus_10`
8. บันทึกทุกอย่างลง `data-template.csv` แล้วทำคนต่อไป

> เคล็ดลับกันอคติ: อย่าบอกผู้เข้าร่วมว่า "เราหวังว่าแอปจะชนะ" ให้เขาทำตามธรรมชาติ

## 4. วิเคราะห์ผล (สคริปต์รันได้จริง)

กรอก CSV เสร็จแล้วรัน:

```sh
node experiments/app-usability/analyze.mjs experiments/app-usability/data-template.csv
```

สคริปต์จะพิมพ์ให้:
- ค่าเฉลี่ย ± ส่วนเบี่ยงเบนมาตรฐาน (SD) ของเวลาและ error ทั้งสองวิธี
- **Paired t-test** (t, องศาอิสระ df, และผลว่ามีนัยสำคัญที่ระดับ 0.05 ไหม) สำหรับเวลาและ error
- **คะแนน SUS เฉลี่ย** พร้อมการตีความ (>68 = ดีกว่าค่าเฉลี่ยระบบทั่วไป)
- ไฟล์ `usability-summary.csv` สำหรับเอาไปทำกราฟแท่งใน Excel/Google Sheets

### การอ่านผล paired t-test (เขียนในเล่ม)
- ถ้า **|t| > ค่าวิกฤต** (สคริปต์เทียบให้แล้ว) → ความต่างของสองวิธี **มีนัยสำคัญทางสถิติ** ไม่ได้เกิดจากความบังเอิญ
- เขียนสรุปเช่น: *"ผู้ใช้บันทึกด้วยแอปเฉลี่ย 62.4 ± 8.1 วินาที เทียบกับสเปรดชีต 118.7 ± 15.3 วินาที (paired t = 9.82, df = 14, p < 0.05) แสดงว่าแอปเร็วกว่าอย่างมีนัยสำคัญ"*

## 5. โบนัส — วัด "ความน่าเชื่อถือของระบบ" (Reliability) ได้ด้วยตัวเลขจริง

นอกจากทดลองกับคน โครงงานซอฟต์แวร์ควรรายงาน **ความถูกต้องของโปรแกรม** ด้วย ในโปรเจกต์นี้มีอยู่แล้ว:

```sh
npx vitest run        # ชุดทดสอบอัตโนมัติ (unit tests) — รายงานจำนวนผ่าน/ทั้งหมด
```

เอาตัวเลข "ผ่าน X จาก Y เคส (คิดเป็น Z%)" ใส่เป็นตัวชี้วัดความน่าเชื่อถือของระบบในเล่มได้ (เช่น ครอบคลุมการแปลงวันที่ พ.ศ., การจับกลุ่มค่าใช้จ่าย, การอ่านสลิป QR, การนำเข้า Excel ฯลฯ)
